Persistent power-up counts bypass affordability
Verwerfen
getPowerUpCount counts persistent inventory even when persistentPowerUpsEnabled is false. canAffordPowerUp then treats disabled saved counts as usable stock and skips coin or move affordability, so hints can suggest power-up moves the player cannot actually use and the no-moves check can stay alive incorrectly.

Candidate evaluations leak user-swap state
Verwerfen
findBestSwap restores isUserSwap and lastSwapPosition only after all candidates finish. Since swap-like evaluation sets both, later hammer and halver candidates, plus cascade rounds, can run with stale user-swap context from an unrelated candidate. The real processor clears isUserSwap after the first match detection, so this can inflate scores and highlight impossible match groups.

Hammer simulation removes the wrong blocked cells
Verwerfen
evaluateHammerOnBlocked always nulls only the selected cell before collapsing.
